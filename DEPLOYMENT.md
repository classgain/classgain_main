# Production deployment

This repository is prepared for three Vercel projects, one Render web service, and MongoDB Atlas.

## Production folder structure

```text
what-next-project/
|-- client/
|   |-- .env.production
|   |-- vercel.json
|   `-- dist/                 # generated by npm run build
|-- seller/
|   |-- .env.production
|   |-- vercel.json
|   `-- dist/
|-- admin/
|   |-- .env.production
|   |-- vercel.json
|   `-- dist/
|-- server/
|   |-- src/
|   |-- uploads/              # local/static upload path
|   |-- .env.example
|   `-- package.json
|-- render.yaml
`-- DEPLOYMENT.md
```

## 1. Prepare MongoDB Atlas

1. Create an Atlas cluster and a database user with a unique password.
2. Add Render's outbound IP ranges to the Atlas IP access list.
3. Copy the Atlas connection string and include a database name in it.
4. Store the connection string only in Render as `MONGO_URI`. Never put it in a frontend variable or commit it.

Example shape:

```text
mongodb+srv://<username>:<password>@<cluster-host>/<database>?retryWrites=true&w=majority
```

If a real credential has ever appeared in an example file or commit, rotate the Atlas password before deployment.

## 2. Deploy the API to Render

The root `render.yaml` is a Render Blueprint. Create a new Blueprint from this repository and enter the variables marked `sync: false`.

Render settings:

| Setting | Value |
| --- | --- |
| Service type | Web Service |
| Runtime | Node |
| Root directory | `server` |
| Build command | `npm ci --omit=dev` |
| Start command | `npm start` |
| Health check | `/api/health` |

Required Render environment variables:

```dotenv
MONGO_URI=mongodb+srv://...
FRONTEND_URLS=https://your-client.vercel.app,https://your-seller.vercel.app,https://your-admin.vercel.app
CLIENT_URL=https://your-client.vercel.app
SELLER_URL=https://your-seller.vercel.app
ADMIN_URL=https://your-admin.vercel.app
JWT_SECRET=<at-least-32-random-characters>
ADMIN_API_KEY=<separate-strong-random-key>
NODE_ENV=production
```

`FRONTEND_URLS` may contain only the frontend deployments that currently exist. The
individual `CLIENT_URL`, `SELLER_URL`, and `ADMIN_URL` variables are optional aliases,
so an API deployment no longer fails merely because the seller or admin frontend has
not been deployed yet. Configure at least one origin by either method.

Render supplies `PORT`; the server falls back to `5000` outside Render. `CLIENT_URL`, `SELLER_URL`, and `ADMIN_URL` must be HTTPS origins without a trailing path. A variable may contain comma-separated origins if a custom domain must also be allowed.

After deployment, verify:

```text
https://your-render-service.onrender.com/api/health
```

The liveness endpoint returns HTTP 200 when Express is online and reports the database
state. Use `/api/ready` to verify the complete API: it returns HTTP 200 only when
MongoDB is connected, or HTTP 503 with `database: "disconnected"`. The server retries
MongoDB automatically, so a temporary Atlas or DNS outage no longer leaves Render
unreachable.

## 3. Deploy the three frontends to Vercel

Create three separate Vercel projects from the same repository.

| Vercel project | Root directory | Build command | Output directory |
| --- | --- | --- | --- |
| Client | `client` | `npm run build` | `dist` |
| Seller | `seller` | `npm run build` | `dist` |
| Admin | `admin` | `npm run build` | `dist` |

For each project, add this Production environment variable and redeploy:

```dotenv
VITE_API_URL=https://your-render-service.onrender.com/api
```

The Seller project also needs `VITE_CLIENT_URL=https://your-client.vercel.app`. The Admin project needs `VITE_CLIENT_URL=https://your-client.vercel.app`, `VITE_SELLER_URL=https://your-seller.vercel.app`, and (for the currently implemented counselling administration endpoints) `VITE_ADMIN_API_KEY` with the same value as Render's `ADMIN_API_KEY`.

`VITE_ADMIN_API_KEY` is embedded in the browser bundle and is therefore only a temporary access control. Protect the Admin Vercel project from public access and replace this key flow with authenticated admin accounts before exposing it to untrusted users.

Do not add secrets to any `VITE_*` variable; Vite embeds these values in the browser bundle. The API URL is public and safe to expose.

Each `vercel.json` builds with `npm ci`, publishes `dist`, disables stale caching for `index.html`, and rewrites all application routes to `index.html` so React Router refreshes and deep links work.

## 4. Finalize CORS

After Vercel assigns the three production domains:

1. Replace `CLIENT_URL`, `SELLER_URL`, and `ADMIN_URL` in Render with the exact domains.
2. Redeploy or restart the Render service.
3. Open each frontend and confirm API requests succeed.

Vercel preview deployments use different domains. Add the exact preview origins to the appropriate comma-separated Render variable only when those previews need API access. The production API intentionally does not allow arbitrary `*.vercel.app` origins.

## Production checks

- Visit a nested React route directly and refresh it; it should still render.
- Register and log in as a student.
- Register and log in as an education partner.
- Confirm the admin frontend can list and update education centers.
- Upload representative media within the configured `JSON_BODY_LIMIT` (10 MB by default).
- Confirm `/api/health` returns HTTP 200.
- Confirm an unknown origin receives a CORS rejection.
- Confirm secrets exist only in Atlas, Render, or local ignored `.env` files.

## Upload storage note

The API serves files from `/uploads` with production caching and safe static-file headers. Render's ordinary filesystem is ephemeral, so files written there can disappear on a redeploy or restart. The current application primarily stores uploaded image data and external video URLs in MongoDB. If disk-based uploads are added later, use a persistent Render disk or object storage such as Cloudinary or S3 and store only the resulting URL in MongoDB.

## Security included

- Helmet security headers and disabled Express signature
- Strict production CORS for the three frontend origins
- General and authentication-specific rate limits
- Gzip response compression
- JSON body size and shape checks, including MongoDB operator-key rejection
- Environment-only MongoDB and JWT secrets
- Render proxy trust and graceful shutdown
- Structured production request and error logs
- Production database-inspection endpoint disabled

The current admin moderation routes retain the application's existing access model. Before giving the admin URL to untrusted users, add authenticated admin accounts or protect the admin deployment and API with an identity-aware access layer.
